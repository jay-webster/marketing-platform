# Implementation Plan: Content Sync & Ingestion Pipeline

**Branch**: `006-content-sync-ingest` | **Date**: 2026-03-14 | **Spec**: [spec.md](./spec.md)

---

## Summary

Wire the connected GitHub repository as the canonical content store. Two paths: (1) a sync engine that walks configured folders via GitHub REST API, upserts `synced_documents`, and queues KB indexing for changed files; (2) an extended file-upload pipeline that generates Markdown, commits to a branch, opens a PR, and lets the admin merge or reject in-app. A new content browser endpoint exposes indexed documents to all roles. Folder management (add/remove) is added inline to the existing GitHub settings page. Email notifications fire on PR merge/close.

---

## Technical Context

**Language/Version**: Python 3.11 / FastAPI (backend), Next.js 15 App Router (frontend)
**Primary Dependencies**: SQLAlchemy async, httpx, Alembic, aiosmtplib, Anthropic SDK, Voyage AI (all existing)
**Storage**: PostgreSQL 16 + pgvector (primary), GCS (transient file staging)
**Testing**: pytest + pytest-asyncio
**Target Platform**: GCP GKE (stateless container)
**Performance Goals**: Sync 100 `.md` files < 60 seconds; PR creation < 30 seconds after worker picks up job
**Constraints**: No local filesystem writes (CONSTITUTION §Stateless Services); all GitHub interaction via REST API (httpx), not git subprocess
**Scale/Scope**: Single-tenant installation; hundreds of `.md` files per repo; tens of concurrent ingestion jobs

## Constitution Check

| Principle | Status | Notes |
|---|---|---|
| **AUTH_SAFE** | Pass | All new endpoints use `require_role()` or `get_current_user`. `/content` requires any authenticated session. |
| **DRY** | Pass | Sync uses `utils/github_api.py` (extended). Email reuses `utils/email.py`. Queue worker extends `utils/queue.py`. Indexer reused for synced docs. |
| **NON_BLOCKING** | Pass | No local filesystem writes. GitHub API calls use httpx async. Sync runs in a background asyncio task. PR creation in existing worker pool. |
| **STATELESS** | Pass | No local volumes. Sync reads files from GitHub REST API only (no git clone). Transient uploads stay in GCS until PR creation. |
| **IDEMPOTENT** | Pass | Sync uses UPSERT on `(connection_id, repo_path)`. Re-syncing same SHA is a no-op. |
| **AUDIT** | Pass | `write_audit()` called for sync trigger, PR created, PR merged, PR closed/rejected, folder added/removed. |
| **ENVIRONMENT** | Pass | All new config via env vars: `SYNC_INTERVAL_HOURS`, `GITHUB_MERGE_METHOD`. |

**Note on `utils/github_client.py`**: The existing git-clone-based client is **not used** in this feature. It relies on local filesystem writes incompatible with stateless K8s pods. The new sync path uses `utils/github_api.py` (httpx REST API only), which is already used for scaffolding and is fully stateless.

## Key Implementation Details

### GitHub API Extensions (utils/github_api.py)

New functions, all async httpx, all raising `GitHubUnavailableError` on timeout/5xx:

| Function | Purpose |
|---|---|
| `get_default_branch(url, token)` | GET /repos/{o}/{r} → `default_branch` string |
| `get_repo_tree(url, token, ref)` | GET /repos/{o}/{r}/git/trees/{ref}?recursive=1 → list of `{path, sha, type}` |
| `get_file_content(url, token, path, ref=None)` | GET /repos/{o}/{r}/contents/{path} → `{content: str (decoded), sha: str}` |
| `create_branch(url, token, name, from_sha)` | POST /repos/{o}/{r}/git/refs |
| `commit_file(url, token, path, content, message, branch, file_sha=None)` | PUT /repos/{o}/{r}/contents/{path} |
| `delete_file(url, token, path, file_sha, message, branch)` | DELETE /repos/{o}/{r}/contents/{path} |
| `create_pr(url, token, title, head, base, body="")` | POST /repos/{o}/{r}/pulls → `{number, url}` |
| `merge_pr(url, token, pr_number, method="merge")` | PUT /repos/{o}/{r}/pulls/{n}/merge |
| `close_pr(url, token, pr_number)` | PATCH /repos/{o}/{r}/pulls/{n} state=closed |

### Sync Orchestration (utils/sync.py)

```
run_sync(connection_id, triggered_by, trigger_type):
  1. Check for in_progress SyncRun → raise if found
  2. Create SyncRun(outcome="in_progress")
  3. Decrypt token from GitHubConnection
  4. get_default_branch → cache in connection.default_branch
  5. get_repo_tree(ref=default_branch)
  6. Filter: type=blob, path ends .md, path starts with any configured folder
  7. Load existing synced_documents for this connection
  8. For each tree file:
     - if content_sha unchanged → files_unchanged++
     - if new or changed → get_file_content, upsert SyncedDocument, queue KB indexing
  9. For each DB doc not in tree → set is_active=False, remove KnowledgeBaseDocument
  10. Update SyncRun(outcome="success", finished_at, counts)
  11. Update GitHubConnection.last_synced_at
```

### Extended process_document (utils/queue.py)

After Claude Markdown structuring, the worker continues:
1. Get active GitHub connection + decrypt token
2. `create_branch` with name `ingest/{sanitized_filename}-{unix_ts}`
3. `commit_file` to `{destination_folder}/{filename}.md` on branch
4. `create_pr` with title = original filename
5. Update `ingestion_document.github_branch`, `github_pr_number`, `github_pr_url`
6. Set `processing_status = "pr_open"`
7. Delete GCS file (post-PR, not post-markdown)

### Indexer Source Dispatch (utils/indexer.py)

```python
# In index_document():
if kb_doc.synced_document_id:
    synced = await db.get(SyncedDocument, kb_doc.synced_document_id)
    markdown = synced.raw_content
    metadata = {"title": synced.title, "repo_path": synced.repo_path, "folder": synced.folder}
else:
    # existing ProcessedDocument path
    proc_doc = await db.get(ProcessedDocument, kb_doc.processed_document_id)
    markdown = proc_doc.structured_content
    metadata = proc_doc.metadata or {}
```

### PR Merge Flow (src/api/ingestion.py)

```
POST /ingestion/documents/{doc_id}/pr/merge:
1. Load doc, verify status=pr_open
2. Decrypt token
3. If destination_folder changed from committed path:
   a. get_file_content(old_path, ref=branch) to get file SHA
   b. delete_file(old_path, branch)
   c. commit_file(new_path, branch)
4. merge_pr(pr_number)
5. Update doc: status=merged, destination_folder=final_folder
6. Trigger sync: call run_sync() in background (asyncio.create_task)
7. Send notification email (best-effort)
8. write_audit("ingestion_pr_merged")
```

### Scheduled Sync Loop (utils/queue.py)

```python
async def _scheduled_sync_loop():
    while True:
        hours = int(os.environ.get("SYNC_INTERVAL_HOURS", "24"))
        await asyncio.sleep(hours * 3600)
        try:
            conn = await _get_active_connection()
            if conn:
                await run_sync(conn.id, triggered_by=None, trigger_type="scheduled")
        except Exception:
            logger.exception("Scheduled sync failed")

async def start_sync_scheduler():
    asyncio.create_task(_scheduled_sync_loop())
```

### Frontend: SyncCard Component

- Shows: last sync time, outcome badge, file counts from last run
- "Sync Now" button → `POST /api/v1/github/sync` → polls `GET /api/v1/github/sync/status` every 3s while `outcome="in_progress"`
- Expandable "View History" accordion showing last 5 runs

### Frontend: FolderManager Component

- Displays configured folders as a list with Remove buttons
- "Add Folder" input with validation (no leading/trailing slash, no `..`)
- Add calls `POST /api/v1/github/config/folders`
- Remove calls `DELETE /api/v1/github/config/folders/{name}` (URL-encodes the path)
- Both invalidate the `["github-config"]` react-query key on success

### Frontend: PRReviewCard Component

- Reads `GET /api/v1/ingestion/documents/{docId}/pr`
- Left column: source filename, submitter, original folder
- Main area: rendered Markdown preview (react-markdown or similar)
- Right column: folder selector (Select from configured folders), Merge button, Reject button
- Merge → `POST .../pr/merge` with selected folder → success toast → redirect to /ingestion
- Reject → confirmation dialog → `POST .../pr/close` → success toast → redirect to /ingestion
